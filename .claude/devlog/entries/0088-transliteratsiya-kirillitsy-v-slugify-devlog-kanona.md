# Транслитерация кириллицы в slugify devlog-канона

## Контекст
Оператор нашёл латентный баг канона: `slugify()` в `rebuild-index.py` выбрасывал всё,
кроме `[a-z0-9]`, поэтому чисто русский title давал пустой slug и файл не проходил
валидацию. Собственный пример из `SKILL.md` («Добавлена фильтрация по ключевым словам»)
был невалиден по этому же правилу. Записи проходили только если в title случайно были
латинские токены. Из двух вариантов (строка-гайд «хотя бы один латинский термин» vs
транслитерация в коде) выбран механический — guard в коде надёжнее prompt-compliance.

## Изменения
- `slugify()`: транслитерация кириллица→латиница перед фильтром (`ж→zh`, `щ→shch`, …);
  `slugify_legacy()` сохраняет старое поведение, валидация принимает оба slug'а —
  существующие 87 entries фабрики валидны без переименований.
- Синхронизированы обе копии: канон `~/.claude/skills/devlog/` и фабрика
  `.claude/devlog/rebuild-index.py`; `SKILL.md` (канон + фабрика) — шаг транслитерации
  задокументирован.
- Регрессионные тесты `~/.claude/skills/devlog/test_rebuild_index.py` (5 шт., RED→GREEN):
  чистая кириллица, mixed title, legacy-имена, отклонение неверного slug'а.

## Verify
RED воспроизведён (slug='' на примере из SKILL.md), после фикса 5 passed; прогон обоих
скриптов на реальном devlog фабрики — 87 entries, 0 ошибок. Plugin `claude-code-harness`
v1.3.0 slugify не содержит — не затронут. Эта запись — дог-фуд фикса: title чисто
русский, slug транслитерирован.
