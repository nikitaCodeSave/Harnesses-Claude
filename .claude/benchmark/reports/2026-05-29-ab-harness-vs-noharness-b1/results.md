# A/B: global practice layer (with-harness vs without) — probe (builds=1)

date: 2026-05-29
agent-under-test: `claude --print` (sonnet-4-6) builds an NL→SQL analyst
toggle: global `~/.claude` practice layer ON (harness) vs OFF (noharness, bare config + creds)
judge: text2sql v2 (12 tasks, live Oracle XE `client_product_bench`, ollama Qwen3-30B for inference)
builds per arm: 1 (quick probe — NOT statistically significant)

## Result
| Arm | v2 gate_pass | stable |
|---|---|---|
| harness | **4/12** | 12/12 |
| noharness | **5/12** | 12/12 |
| **delta (harness − noharness)** | **−1** | — |

Per-task: both pass T2, T7, G1, G2; noharness also passes T10. Both fail T1, T3,
T4, T5, T6, T8, T9 (the domain-trap tasks). Full data: `report.json`.

## Honest interpretation
- **No harness advantage on this single-shot build.** delta = −1 at **n=1** is
  **noise**, not a real negative — the claude build is stochastic and a single
  build per arm has no significance (the v2 oracle itself is deterministic, 5/5/5,
  but the build is not).
- **Both arms built near-identical impls**: a `nl2sql/` package (agent/llm/parser/
  validator) + 3 test files + requirements + solve.py. The methodology pillar (§5
  test-quality) produced **no visible delta** in structure or tests — because Opus
  4.8 already writes structured, tested code natively. (noharness was even slightly
  richer: +schema.py, +conftest.py.)
- The actual SQL is generated by the **same ollama model** in both arms, so the
  model dominates the score; claude's wrapper engineering moved it marginally.

## What this confirms (the central thesis)
Consistent with devlog #52 and the foundational principle: **under Opus 4.8 the
harness's value is NOT single-shot output quality** (the model is already
excellent) **but the continuity/practice delta across sessions** — which a
single-shot A/B cannot capture by construction. This probe is the rigorous,
honest measurement that demonstrates it on a quality oracle (not pass/fail).

## Caveats / next
- n=1 → run `--builds 3+` per arm for any directional claim (delta is within noise).
- The **continuity pillar (#54) is structurally not exercised** by a single-shot
  build → the open task remains a **multi-session** scenario (session N builds on
  devlog/progress 1..N-1) judged by v2. That is where harness value should appear,
  if anywhere.
- Both arms fail the domain traps (T1/T3/T4/T5/T6/T8/T9): a thin ollama wrapper
  without the project's full domain prompt doesn't encode two-step AVG, CREATE_DT
  lifecycle, etc. — same ceiling as the #59 thin-hint baseline.

## Qualitative code comparison (depth / professionalism)
The gate_pass metric (4 vs 5) only sees SQL correctness. Operator hypothesis: the
**harness** arm might be more professional in deep-development terms. A read-only
engineering review of both preserved codebases (`build-harness/`, `build-noharness/`),
verified against the source, says **the opposite — marginally**:

| Axis | Winner | Evidence |
|---|---|---|
| Architecture | tie (slight noharness) | noharness `NL2SQLAgent` holds message state for repair (`agent.py:13-22`); harness rebuilds a string suffix |
| Test depth | **noharness** | asserts consequences (rejects SUM-on-balance, unbalanced parens, missing table); harness tests largely restate its own 3 checks |
| Validation / errors | **noharness** | balance-SUM domain guard (`validator.py:41-54`), word-boundary DML keywords (catches stacked `DROP`), parens balance, env/empty-input preconditions in `solve.py`. harness validator = SELECT-start / start-anchored DML / FROM only |
| Prompt engineering | tie | both schema-aware with AVG-vs-SUM, FETCH FIRST, NO_SQL, few-shot |
| Robustness | **noharness** | strips `<think>…</think>` (reasoning models), httpx→urllib fallback |
| Typing / docs | tie | harness tighter types (`str \| None`); noharness fuller docstrings |
| Domain rules in CODE | **noharness** | the headline trap (two-step AVG) is an executable+tested guard, not just prose; harness encodes it only in the prompt |
| **Counter (harness better)** | harness | parser auto-rewrites `LIMIT → FETCH FIRST` (`parser.py:40`); a call-count efficiency test; precise type hints |

**Honest verdict: the hypothesis is NOT supported on this n=1 run.** The arm *without*
the global practice layer produced the deeper validation and more behavior-driven
tests; the harness arm's only clear depth win was the LIMIT→FETCH-FIRST self-heal.
The gap is **marginal and within single-sample noise** — both are strong,
architecturally near-identical builds. This is actually *consistent* with the
foundational principle: under Opus 4.8 the model already writes this depth natively,
so a one-shot build doesn't reveal a harness advantage.

What the 4-vs-5 metric misses (and the review surfaced): the executable domain guard,
injection hardening, reasoning-token robustness, env preconditions, and behaviour-vs-
restatement test quality — all gate-invisible, and here they tilt slightly to noharness.

**Caveat:** depth is exactly what could vary build-to-build. A real test of the depth
hypothesis needs `--builds N` per arm scored on a code-quality rubric (not just
gate_pass) — n=1 cannot settle it. Both codebases preserved under `build-*/` for audit.
