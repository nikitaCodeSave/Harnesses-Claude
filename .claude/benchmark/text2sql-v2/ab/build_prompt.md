Построй небольшую систему NL→SQL для аналитики по таблице Oracle `client_product`.

## Что это за данные
`client_product` — помесячный снимок по корпоративному клиенту банка (одна строка =
клиент × месяц). Ключевые колонки:
- `MONTH_DT` (DATE) — дата снимка. У закрытого месяца = конец месяца (EOM); у текущего
  (незакрытого) месяца дата НЕ конец месяца — это последняя дата выгрузки.
- `CREATE_DT`, `CLOSE_DT` (DATE) — даты открытия/закрытия клиента (события, не снимок).
- `INN` (NUMBER), `ORGANIZATION_NM`, `GROUP_NM`, `SEG` — идентификация и разрезы.
- Балансы: `CA_LCY_SUM`, `CA_LCY_RUB_SUM`, `DEP_SUM` (за период >1 месяца усредняются
  по месяцам, не суммируются).
- FX: `FX_VOLUME_RUB`, `FX_CNT`, `FX_MARGIN_RUB` (+ EUR/CNY-разбивки).
- Доход: `PNL_SUM` (итог = сумма компонентов + прочее); компоненты `PL_CA`, `PL_DEP`,
  `PL_VK`, `PL_VK_OUT`, `VED_SUM`, `FX_MARGIN_RUB`.
- ВЭД: `VED_VOL`. Активность: `ACT_1M`, `ACT_3M` (0/1).
СУБД — Oracle (используй `FETCH FIRST`/подзапросы, `COUNT(DISTINCT ...)`).

## Задача
Система получает аналитический вопрос на русском и возвращает ОДИН корректный Oracle
SQL-запрос (SELECT) по `client_product`, отвечающий на вопрос. Запрос НЕ исполняй —
только верни его текст. Если на вопрос нельзя ответить из этой таблицы (нет таких
данных, или вопрос не про эти данные) — верни строго `NO_SQL`.

## Как генерировать SQL — через LLM-инференс на локальной ollama
Это агент NL→SQL поверх LLM (как реальный проект). Генерируй SQL, обращаясь к
**локальной ollama** по OpenAI-совместимому API. Конфиг — из переменных окружения
(они уже заданы, не хардкодь):
- `AI_ANALYST_API_URL` (например `http://localhost:11434/v1`)
- `AI_ANALYST_API_KEY` (любая непустая строка, напр. `ollama`)
- `AI_ANALYST_SQL_MODEL` (имя модели)
Инженерная ценность — в обвязке вокруг модели: хороший промпт со схемой и доменными
правилами, парсинг ответа, валидация/починка SQL, обработка ошибок. Думай о
корректности РЕЗУЛЬТАТА (правильные агрегаты/периоды), не только синтаксиса.

**Технические ограничения:**
- HTTP к ollama делай через `httpx` ИЛИ стандартный `urllib` (оба доступны).
  Пакета `openai` в окружении НЕТ — не используй его (или поставь сам в requirements,
  но проще httpx/urllib). **НИКОГДА не используй Anthropic API / пакет `anthropic`**
  (ключа нет, это запрещено).
- SQL НЕ исполняй и к БД НЕ подключайся — это делает бенчмарк. Верни только текст SQL.

## Контракт интерфейса (ОБЯЗАТЕЛЬНО, иначе бенчмарк не сможет тебя оценить)
Создай в корне рабочей директории файл **`solve.py`** такой, что:
```
python3 solve.py "<вопрос на русском>"
```
печатает в stdout РОВНО SQL-запрос (или `NO_SQL`, если данных в таблице нет) и ничего
лишнего (диагностику — в stderr). Без интерактива. Точка входа — `solve.py` в корне
(логику можешь раскладывать по модулям).

## Качество
Покрой ключевые сценарии (агрегаты по периодам, средние остатки, структура дохода,
сравнение периодов, lifecycle по датам, активность, топ-N, отказ на нерелевантное)
тестами рядом с кодом и убедись, что они проходят. Думай о корректности результата,
не только о синтаксисе SQL.
